📁 File System — ทุกอย่างคือไฟล์

🔑 Permissions — ใครทำอะไรได้

เข้าใจ Permission String

-rwxr-xr-- 1 deploy www-data 4096 Mar 7 10:30 app.py
│└┬┘└┬┘└┬┘   └──┬─┘ └───┬──┘
│ │  │  │       │       │
│ │  │  │       │       └── Group
│ │  │  │       └── Owner
│ │  │  └── Others (r-- = read only)
│ │  └── Group  (r-x = read + execute)
│ └── Owner  (rwx = read + write + execute)
└── Type (- = file, d = directory, l = symlink)

Permission Numbers

r (read)    = 4
w (write)   = 2
x (execute) = 1

ตัวอย่าง:
rwx = 4+2+1 = 7
r-x = 4+0+1 = 5
r-- = 4+0+0 = 4

chmod 755 = rwxr-xr-x  (owner: all, group+others: read+execute)
chmod 644 = rw-r--r--  (owner: read+write, group+others: read only)
chmod 600 = rw-------  (owner only — ใช้กับ SSH keys)
chmod 777 = rwxrwxrwx  (ทุกคนทำได้หมด — ❌ อย่าใช้ใน production!)

คำสั่งจัดการ Permissions

# เปลี่ยน permission
chmod 755 script.sh              # ตัวเลข
chmod u+x script.sh              # เพิ่ม execute ให้ owner
chmod g-w config.yml             # ลบ write จาก group
chmod o-rwx secret.key           # ลบทุกอย่างจาก others
chmod -R 755 /var/www/           # recursive (ทั้ง directory)

# เปลี่ยน owner
chown deploy:www-data app.py     # owner: deploy, group: www-data
chown -R deploy:deploy /opt/myapp/   # recursive

# เปลี่ยน group
chgrp docker /var/run/docker.sock

# ดู permissions
ls -la                           # list all with permissions
stat file.txt                    # detailed file info

Special Permissions

# SUID — รันด้วยสิทธิ์ของ owner (ไม่ใช่ผู้รัน)
chmod u+s /usr/bin/passwd        # -rwsr-xr-x
# → user ธรรมดารัน passwd ได้ แม้ต้องแก้ /etc/shadow

# SGID — รันด้วย group ของ file/directory
chmod g+s /shared/project/       # ไฟล์ใหม่ inherit group

# Sticky bit — ลบได้เฉพาะ owner
chmod +t /tmp/                   # drwxrwxrwt
# → ทุกคนเขียนได้ แต่ลบได้เฉพาะไฟล์ตัวเอง

ดู Processes

# แบบ snapshot
ps aux                           # ทุก process
ps aux | grep nginx              # หา nginx

# แบบ real-time
top                              # classic
htop                             # สวยกว่า (ต้อง install)
btop                             # สวยที่สุด (modern)

# ดู process เฉพาะ

pgrep -la python                 # หา python processes + command
lsof -i :8080                    # ใครใช้ port 8080 อยู่

จัดการ Processes

# ส่ง signal
kill <PID>                       # SIGTERM (ขอให้หยุดดีๆ)
kill -9 <PID>                    # SIGKILL (บังคับหยุด — ไม่ cleanup)
kill -HUP <PID>                  # reload config (Nginx, Apache)
killall python                   # kill ทุก python process
pkill -f "gunicorn"              # kill by pattern

# Signals ที่ DevOps ต้องรู้
# SIGTERM (15) — "กรุณาหยุด" → process cleanup แล้วจบ
# SIGKILL (9)  — "หยุดเดี๋ยวนี้" → ฆ่าทันที ไม่ cleanup
# SIGHUP  (1)  — "reload config" → ใช้กับ daemon
# SIGINT  (2)  — Ctrl+C → interrupt

# Background / Foreground
./long-task.sh &                 # รันใน background
jobs                             # ดู background jobs
fg %1                            # เอากลับมา foreground
bg %1                            # ส่งกลับ background
nohup ./script.sh &              # รันต่อแม้ logout
disown %1                        # ตัดจาก terminal

🔧 systemd — จัดการ Services

systemd = init system ของ Linux ยุคใหม่ ใช้จัดการ services ทั้งหมด

คำสั่งพื้นฐาน

# จัดการ service
sudo systemctl start nginx       # เริ่ม
sudo systemctl stop nginx        # หยุด
sudo systemctl restart nginx     # restart
sudo systemctl reload nginx      # reload config (ไม่ restart)
sudo systemctl enable nginx      # เปิดตอน boot
sudo systemctl disable nginx     # ไม่เปิดตอน boot

# ดูสถานะ
systemctl status nginx           # สถานะ + log ล่าสุด
systemctl is-active nginx        # active or inactive
systemctl is-enabled nginx       # enabled or disabled
systemctl list-units --type=service  # ทุก service
systemctl list-units --failed    # service ที่พัง

สร้าง Service ของตัวเอง

# /etc/systemd/system/myapp.service
[Unit]
Description=My Python Application
After=network.target postgresql.service
Wants=postgresql.service

[Service]
Type=simple
User=deploy
Group=www-data
WorkingDirectory=/opt/myapp
Environment=NODE_ENV=production
Environment=PORT=8080
ExecStart=/opt/myapp/.venv/bin/python app.py
ExecReload=/bin/kill -HUP $MAINPID
Restart=always
RestartSec=5
StartLimitBurst=5
StartLimitIntervalSec=60

# Security hardening
NoNewPrivileges=yes
ProtectSystem=strict
ProtectHome=yes
ReadWritePaths=/opt/myapp/data /var/log/myapp

# Resource limits
MemoryMax=512M
CPUQuota=200%

# Logging
StandardOutput=journal
StandardError=journal
SyslogIdentifier=myapp

[Install]
WantedBy=multi-user.target
# ใช้งาน
sudo systemctl daemon-reload              # โหลด unit file ใหม่
sudo systemctl enable --now myapp          # enable + start ทันที
sudo journalctl -u myapp -f               # ดู logs real-time
sudo journalctl -u myapp --since "1 hour ago"

systemd Timers (แทน cron)

# /etc/systemd/system/backup.timer
[Unit]
Description=Daily backup timer

[Timer]
OnCalendar=*-*-* 02:00:00        # ทุกวัน ตี 2
Persistent=true                   # รันถ้าพลาดไป (เครื่องปิดอยู่)
RandomizedDelaySec=300            # random delay 0-5 min

[Install]
WantedBy=timers.target
# /etc/systemd/system/backup.service
[Unit]
Description=Daily backup

[Service]
Type=oneshot
ExecStart=/opt/scripts/backup.sh
sudo systemctl enable --now backup.timer
systemctl list-timers             # ดู timers ทั้งหมด

🔐 SSH — Remote Access

SSH Basics

# เชื่อมต่อ
ssh user@server                  # ใช้ password
ssh -i ~/.ssh/mykey.pem user@server  # ใช้ key
ssh -p 2222 user@server          # port อื่น

# สร้าง SSH key pair
ssh-keygen -t ed25519 -C "deploy@myserver"
# → ~/.ssh/id_ed25519      (private — ห้ามแชร์!)
# → ~/.ssh/id_ed25519.pub  (public — ส่งให้ server)

# ส่ง public key ไป server
ssh-copy-id user@server

# Permission ที่ถูกต้อง (สำคัญมาก!)
chmod 700 ~/.ssh/
chmod 600 ~/.ssh/id_ed25519          # private key
chmod 644 ~/.ssh/id_ed25519.pub      # public key
chmod 600 ~/.ssh/authorized_keys

SSH Config (ชีวิตง่ายขึ้น 10 เท่า)

# ~/.ssh/config
Host production
    HostName 203.0.113.50
    User deploy
    IdentityFile ~/.ssh/prod_key
    Port 22

Host staging
    HostName 10.0.1.50
    User deploy
    IdentityFile ~/.ssh/staging_key
    ProxyJump bastion

Host bastion
    HostName 203.0.113.10
    User admin
    IdentityFile ~/.ssh/bastion_key

Host *.internal
    ProxyJump bastion
    User deploy

# ใช้งาน:
# ssh production   แทน  ssh -i ~/.ssh/prod_key deploy@203.0.113.50
# scp file.txt production:/tmp/

SSH Tunneling (Port Forwarding)

# Local forward — เข้าถึง remote service ผ่าน localhost
ssh -L 5432:db-server:5432 bastion
# → localhost:5432 → bastion → db-server:5432
# ใช้เมื่อ: DB อยู่หลัง firewall

# Remote forward — เปิด port ให้ remote เข้า local
ssh -R 8080:localhost:3000 server
# → server:8080 → กลับมาที่เครื่องเรา :3000

# Dynamic (SOCKS proxy)
ssh -D 1080 server
# → ใช้เป็น SOCKS proxy ทั้ง browser

# แบบ background
ssh -fN -L 5432:db:5432 bastion
# -f: background, -N: no command

Hardening SSH

# /etc/ssh/sshd_config — ต้องแก้ใน production!
PermitRootLogin no               # ห้าม SSH เป็น root
PasswordAuthentication no        # ใช้ key เท่านั้น
PubkeyAuthentication yes
MaxAuthTries 3
ClientAliveInterval 300
ClientAliveCountMax 2
AllowUsers deploy admin          # whitelist users
Port 2222                        # เปลี่ยน port (optional)

# Apply
sudo systemctl restart sshd

📝 Shell Scripting

Bash Fundamentals

#!/bin/bash
set -euo pipefail  # ⭐ บรรทัดแรกของทุก script!
# -e: หยุดทันทีถ้า command fail
# -u: error ถ้าใช้ variable ที่ไม่ได้ set
# -o pipefail: pipe fail = script fail

# === Variables ===
NAME="production"
PORT=8080
echo "Deploying to $NAME on port $PORT"

# === Command substitution ===
CURRENT_DATE=$(date +%Y-%m-%d)
FILE_COUNT=$(ls -1 | wc -l)
MY_IP=$(curl -s ifconfig.me)

# === Conditionals ===
if [[ -f /etc/nginx/nginx.conf ]]; then
    echo "Nginx config exists"
elif [[ -d /etc/apache2 ]]; then
    echo "Apache installed instead"
else
    echo "No web server found"
fi

# === Test operators ===
# Files:  -f (exists) -d (dir) -r (readable) -w (writable) -x (executable) -s (not empty)
# Strings: -z (empty) -n (not empty) == !=
# Numbers: -eq -ne -lt -gt -le -ge

# === Loops ===
for server in web1 web2 web3; do
    echo "Deploying to $server..."
    ssh deploy@$server "cd /opt/app && git pull && systemctl restart app"
done

# While loop — wait for health
while ! curl -s http://localhost:8080/health > /dev/null; do
    echo "Waiting for app to start..."
    sleep 2
done
echo "App is healthy!"

# Read file line by line
while IFS= read -r line; do
    echo "Processing: $line"
done < servers.txt

# === Functions ===
deploy() {
    local server=$1
    local version=$2
    echo "Deploying v$version to $server"
    ssh deploy@$server "
        cd /opt/app
        git fetch && git checkout v$version
        pip install -r requirements.txt
        sudo systemctl restart app
    "
}

deploy web1 2.1.0
deploy web2 2.1.0

Practical Deploy Script

#!/bin/bash
# deploy.sh — Simple deployment script
set -euo pipefail

APP_NAME="myapp"
DEPLOY_DIR="/opt/$APP_NAME"
BACKUP_DIR="/opt/backups"
VERSION=${1:?"Usage: $0 <version>"}

echo "🚀 Deploying $APP_NAME v$VERSION"

# 1. Backup
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
cp -r "$DEPLOY_DIR" "$BACKUP_DIR/${APP_NAME}_${TIMESTAMP}"

# 2. Pull new version
cd "$DEPLOY_DIR"
git fetch --all
git checkout "v$VERSION"

# 3. Install dependencies
pip install -r requirements.txt --quiet

# 4. Run migrations
python manage.py migrate --noinput

# 5. Restart
sudo systemctl restart "$APP_NAME"

# 6. Health check
sleep 5
if curl -sf http://localhost:8080/health > /dev/null; then
    echo "✅ Deploy successful! v$VERSION is live"
else
    echo "❌ Health check failed! Rolling back..."
    sudo systemctl stop "$APP_NAME"
    rm -rf "$DEPLOY_DIR"
    cp -r "$BACKUP_DIR/${APP_NAME}_${TIMESTAMP}" "$DEPLOY_DIR"
    sudo systemctl start "$APP_NAME"
    echo "⏪ Rolled back to previous version"
    exit 1
fi

# 7. Cleanup old backups (keep last 5)
ls -dt "$BACKUP_DIR"/${APP_NAME}_* | tail -n +6 | xargs rm -rf
echo "🧹 Old backups cleaned"

Health Check Script

#!/bin/bash
# health-check.sh — Monitor multiple services
set -euo pipefail

SERVICES=(
    "http://web1:8080/health"
    "http://web2:8080/health"
    "http://api:3000/health"
)
SLACK_WEBHOOK="https://hooks.slack.com/services/xxx/yyy/zzz"

for url in "${SERVICES[@]}"; do
    if ! curl -sf --max-time 5 "$url" > /dev/null 2>&1; then
        MESSAGE="🚨 Health check failed: $url"
        echo "$MESSAGE"
        curl -sf -X POST "$SLACK_WEBHOOK" \
            -H 'Content-Type: application/json' \
            -d "{\"text\": \"$MESSAGE\"}"
    fi
done

📅 Cron — Scheduled Tasks

*/5 * * * * flock -n /tmp/health.lock /opt/scripts/health-check.sh

🧰 Pipes & Redirection — พลังของ Shell

# Pipes (|) — output ของคำสั่งก่อน = input ของคำสั่งถัดไป

# → อ่าน log → หาบรรทัด POST → นับจำนวน

# Process substitution — เทียบ config ระหว่าง 2 servers!

📋 Troubleshooting Decision Tree

🔴 ระบบมีปัญหา!
  • เว็บเข้าไม่ได้?
    • ping server → ❌ → เช็ค network/firewall
    • curl localhost:80 → ❌ → systemctl status nginx
    • curl ✅ แต่จากข้างนอก ❌ → ufw status / iptables -L
    • ได้ response แต่ error → tail -f /var/log/nginx/error.log
  • Server ช้า?
    • top → CPU สูง? → ps aux --sort=-%cpu | head
    • free -h → RAM เต็ม? → dmesg | grep -i oom
    • df -h → Disk เต็ม? → find / -name "*.log" -size +100M
    • iostat → Disk I/O สูง? → iotop
  • Service crash?
    • systemctl status myapp
    • journalctl -u myapp --since "10 min ago"
    • dmesg | tail → kernel errors?
  • SSH เข้าไม่ได้?
    • ping → ❌ → server down / network issue
    • ssh -v user@server → ดู debug output
    • Permission denied? → เช็ค key chmod 600 + authorized_keys + sshd_config
    • Connection refused? → systemctl status sshd

🔑 Key Takeaways

  1. File System: /etc = config, /var/log = logs, /proc = system info
  2. Permissions: chmod 755 (dirs), 644 (files), 600 (secrets) — 777 ห้ามใช้!
  3. Processes: ps aux, htop, kill -15 ก่อน -9 เสมอ
  4. systemd: systemctl start/stop/enable/status — เขียน .service file เอง
  5. SSH: ใช้ key-based auth เท่านั้น, ปิด root login, เขียน ~/.ssh/config
  6. Scripting: set -euo pipefail ทุก script, ใช้ flock กัน overlap
  7. Pipes: | คือพลังของ Unix — ต่อ commands เข้าด้วยกัน
  8. Troubleshoot: CPU → RAM → Disk → Network (เช็คเรียงจากบ่อยสุด)
