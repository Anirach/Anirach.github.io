Beyond Plugins: How I Actually Use OpenClaw as a Production AI Assistant

A year of architecture over plugins — memory systems, healthcare analytics, sub-agent orchestration, and the messy reality of building AI that works.

By Anirach Mingkhwan, Ph.D., KMUTNB · March 19, 2026 · 12 min read

Introduction

If you've spent any time in OpenClaw communities, you've seen the posts: "5 Essential Plugins That Will 10x Your Productivity!" or "860 Integrations in One Click!" The plugin ecosystem has become a race to the bottom — quantity over quality, flash over substance.

I'm here to tell you about the other path. For the past year, I've been running Arthur, a production AI assistant built on OpenClaw that manages my research, healthcare data analytics, memory systems, and academic tooling. Not through a collection of plugins, but through careful architecture.

Arthur runs 24/7 on a VPS, connected via Telegram, handling everything from National Patient Safety dashboards with 4.7 million healthcare records to generating board-level reports in a single evening. This isn't another "productivity hack" post — this is the messy, honest reality of building an AI system that actually works.

The truth is, a well-configured OpenClaw with the right architecture beats any plugin collection. Let me show you how.

The Architecture That Actually Matters

Most people think OpenClaw is about connecting services. That's backwards. OpenClaw is about routing intelligence.

My setup treats OpenClaw as the central gateway that routes requests to the right model based on task complexity:

┌─────────────────────────────────────────────┐
│              OpenClaw Gateway                │
│         (Intelligence Router)               │
├─────────────┬──────────────┬───────────────┤
│  Claude Opus│ Claude Sonnet│   GPT-4o      │
│  Complex    │ Routine      │   Specific    │
│  reasoning  │ tasks, PRs   │   integrations│
│  + review   │ + testing    │   + batch     │
└──────┬──────┴──────┬───────┴───────┬───────┘
       │             │               │
  Orchestrator   Sub-agents     Cost-optimized
  + Code review  + Docs         batch ops

This isn't plugin-driven model selection — it's built into OpenClaw's native model aliases. No "router plugin" needed. Just smart configuration.

Cost control happens through intelligent routing, not through separate cost-monitoring plugins. When Arthur spawns sub-agents for heavy tasks, the main chat stays responsive. When it processes healthcare data, it uses the most cost-effective model that meets security requirements.

The real magic is sub-agent orchestration. Instead of blocking the main conversation while processing complex requests, Arthur spawns background workers. Sub-agents handle document generation, database queries, research compilation — then deliver results directly via Telegram file upload. The main agent never freezes.

This architectural pattern — spawn, work, deliver — eliminates the "typing indicator hell" that plagues most AI assistants.

Memory That Persists Across Sessions (Without a Plugin)

Every AI assistant faces the same problem: they forget everything between sessions. The plugin ecosystem's answer? Install a memory plugin that stores everything in some vendor's cloud.

My approach: build a memory system that actually works.

Arthur's memory system combines three layers:

  1. Hindsight — an open-source knowledge graph running locally
  2. Obsidian vault — human-readable notes with bidirectional sync
  3. Daily memory files — session-by-session learning logs

The bidirectional sync is key: new Obsidian notes auto-ingest into Hindsight every 30 minutes via cron. Hindsight reflections write back to Obsidian as structured notes. Arthur can search semantically across the entire knowledge base.

# Semantic memory search $ python3 tools/memory_search "healthcare accreditation trends" # Auto-reflect on patterns $ python3 tools/memory_reflect "What patterns emerge from recent research?" # Retain new knowledge $ python3 tools/hindsight_helper.py retain "HAI database has 87 tables, 4.7M records"

But here's the hard lesson: bulk ingesting crashed the Docker container 126 times. The solution wasn't more memory or a different plugin — it was gentle batching. 2 items at a time, 3-second delays, 4GB memory limit. Sometimes the best optimization is going slower.

No plugin needed. Just Docker, a Python sync script, and the patience to get it right.

Real Database Integration: Healthcare Analytics

When people talk about "connecting 860 apps," they usually mean surface-level API integrations. Arthur connects to Thailand's Healthcare Accreditation Institute data warehouse — 87 tables, 4.7 million records, real healthcare data that affects real patients.

This isn't "one-click integration." This is careful, secure architecture that respects data sovereignty.

Arthur built a National Patient Safety Dashboard from this data: 1,584 hospitals, 155,000 beds, tracking safety incidents across the entire Thai healthcare system. But every query goes through a mandatory de-identification wrapper:

# NEVER raw psql — always through the wrapper $ python3 tools/ha_query.py "SELECT enrollment_name, COUNT(*) FROM bi_hcr_thip GROUP BY 1" # Wrapper handles: # ✓ VPN connection to secure network # ✓ Query sanitization # ✓ Automatic de-identification (PDPA compliance) # ✓ Result formatting

PDPA compliance means sensitive healthcare data stays local — it never touches cloud APIs. When Arthur generates reports from this data, the analysis happens on-premises, not in some vendor's cloud.

This isn't "connect everything" philosophy. This is thoughtful integration that puts security and compliance first.

Academic Research Toolkit (Built, Not Installed)

Academic publishing has specific requirements: publication-ready charts, precise formatting, editable PDF text. Instead of hunting for plugins, I built Arthur's Academic Charts Toolkit in a single session when I needed it.

Six chart types: Sankey diagrams, 3D scatter plots, heatmaps, grouped bar charts, radar charts, and multi-panel composers. All publication-ready: Arial fonts, 300 DPI resolution, editable PDF text (fonttype 42).

Pure matplotlib — no Chrome dependencies, no Plotly cloud services, runs anywhere. When Arthur generates research visualizations, they're ready for peer review without additional processing.

This toolkit wasn't downloaded from a marketplace. It was designed for my specific research needs: healthcare data visualization, patient safety trend analysis, institutional performance metrics. When you build tools instead of installing them, they fit exactly what you need.

The Document Pipeline Nobody Talks About

Everyone shows you the input side — "Ask Arthur to write a report!" Nobody shows you the output pipeline. How do draft reports become board-level documents? How do research findings become publication-ready papers?

Arthur's document pipeline handles enterprise-grade output: DOCX reports with template systems, automatic table of contents, page numbering, styled tables with consistent formatting. The HAI IT Strategy report — 52 tables, 2,775 words, board-level quality — was generated and iterated in a single evening.

But the real innovation is file delivery. Sub-agents don't just save files locally — they upload to Google Drive with folder conventions and send directly via Telegram. When Arthur completes a research report, you get three things immediately:

  1. 📱 File delivered to your device via Telegram
  2. ☁️ Clean Google Drive link with proper folder organization
  3. 📝 Session log documenting the generation process

No waiting for uploads. No hunting through folders. No wondering which version is current. The pipeline handles distribution automatically.

What I'd Actually Recommend

After a year of production usage, here's what actually matters:

1

Invest in Memory Architecture

Hindsight + daily logs + semantic search beats any memory plugin. Build a system that captures what matters, discards what doesn't, and surfaces insights when you need them.

2

Master Sub-Agent Orchestration

Learn to spawn background workers and keep your main chat responsive. Users don't want to stare at typing indicators for 5 minutes while you generate a report.

3

Build Domain-Specific Tools

A custom wrapper for healthcare data is worth more than 860 generic integrations. When you build tools for your specific domain, they handle edge cases and respect compliance requirements.

4

Self-Healing Patterns

Maintain a .learnings/ directory that captures errors, corrections, and feature gaps. Every mistake becomes institutional knowledge. The system literally learns from its mistakes.

5

Observability (The One Gap)

Tracking sub-agent spawning, model usage costs, memory system performance — this is honestly where plugins might help. Even my "plugin recommendation" is for observability, not core functionality.

The Uncomfortable Truth About Plugin Ecosystems

Plugin ecosystems optimize for developer adoption, not user outcomes. Easy installation drives downloads. Flashy demos drive social media engagement. But production usage reveals different priorities:

Reliability over features. Arthur handles healthcare data that affects patient safety. I need systems that work consistently, not systems that do 860 things poorly.

Integration depth over breadth. One well-integrated database connection with proper security is worth more than surface-level integrations to 100 services.

Maintenance burden. Every plugin is a dependency that can break. Arthur's core memory system has run for 11 months without intervention. How many plugin-based setups can say that?

The uncomfortable truth: the best OpenClaw setups aren't about plugins. They're about architecture.

What Actually Makes a Production AI Assistant

After a year of running Arthur in production, here's what separates hobby projects from real systems:

Structured workspace. SOUL.md defines personality and boundaries. AGENTS.md documents task routing. MEMORY.md captures institutional knowledge. TOOLS.md lists available capabilities. These aren't plugin outputs — these are architectural decisions.

Error recovery patterns. When sub-agents timeout, when database connections fail, when file uploads error — what happens? Arthur logs everything, learns from failures, and builds resilience over time.

Security by design. Healthcare data never touches cloud APIs. De-identification happens automatically. File permissions are restricted. This isn't plugin-configurable security — this is architectural security.

Cost awareness. Smart model routing, efficient sub-agent usage, local processing for sensitive data. Arthur's monthly usage costs haven't exceeded $7 in any month, despite handling 4.7 million healthcare records.

Conclusion

The best OpenClaw setup isn't about plugins — it's about architecture. It's about building systems that work reliably, handle your specific domain requirements, and improve over time through structured learning.

Arthur manages research workflows, healthcare analytics, academic publishing, and institutional reporting because it was designed as a system, not assembled from plugins. The memory architecture persists knowledge across sessions. The security model respects data sovereignty. The document pipeline handles enterprise requirements. The sub-agent orchestration keeps interactions responsive.

Could I have achieved some of this with plugins? Maybe. Would it have been as reliable, secure, and tailored to my needs? Definitely not.

🏗️ Your OpenClaw competitive advantage isn't in your plugin collection

It's in your workspace structure, your architectural decisions, and your willingness to build tools that solve real problems.

Stop collecting plugins. Start building systems.
